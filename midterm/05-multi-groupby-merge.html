Section 5: 多列 Groupby 与 Merge

📋 考点速览

  • 多列 groupby：df.groupby(['col1', 'col2']).agg() —— 产生 MultiIndex，每个唯一组合一行
  • MultiIndex 扁平化：用 .reset_index() 把多层 index 还原回普通列
  • merge 默认是 inner join：左右两边都要有匹配，否则该行消失
  • 预测 merge 行数（最高频考点！）：对每个 key 值，结果行数 = 左边出现次数 × 右边出现次数，再求和
  • 三种 merge 写法：left_on=/right_on=、on=（同名列）、left_index=True/right_index=True（按 index 合并）
🔥 出题套路：每份 midterm 几乎必考一道"merge 后有几行？"的题。掌握"对每个 key 数左右出现次数再相乘求和"这个算法，就能稳拿这分。

📚 知识点

1. 多列 groupby —— groupby([col1, col2])

把 列名 list 传给 .groupby，可以按多列同时分组。结果中每行对应一个唯一组合。

# 例：roster 有 'section' 列（A/B/C/D）和 'first' 列（first name）
# 想知道每个 section 里每个 first name 各有多少人
counts = roster.groupby(['section', 'first']).count()
# counts 的 index 是 MultiIndex (section, first)
# 例如 (A, 'Aayushi')→ 1, (B, 'Andrew')→ 3, ...

核心理解：先按 'section' 分组，每组内再按 'first' 分组 —— 即"子组（subgroup）"，groups within groups。

顺序问题：groupby(['section', 'first']) 与 groupby(['first', 'section']) 内容相同，只是行/列顺序不同。

2. MultiIndex 与 .reset_index()

多列 groupby 之后，结果的 index 是 MultiIndex（多层 index），每行的 index 是一个 tuple，例如 ('A', 'Aayushi')。

用 .reset_index() 把 MultiIndex "压平"，使原来作为 index 的列变回普通列：

flat = roster.groupby(['section', 'first']).count().reset_index()
# 现在 'section' 和 'first' 都是普通列，index 是 0, 1, 2, ...

多列 groupby 的输出按列顺序字母排序（先按第一列字母序，第一列相同时按第二列字母序）。

3. .merge —— 合并两个 DataFrame

用 merge 把两个 DataFrame 按某个 key 拼起来。基本语法：

# 三种写法

# (1) 列名不同
left.merge(right, left_on='col_a', right_on='col_b')

# (2) 列名相同 —— 用 on=
left.merge(right, on='col')

# (3) 用 index 合并
left.merge(right, left_index=True, right_on='col')
left.merge(right, left_index=True, right_index=True)

结果规则：

  • 对左 DataFrame 的每一行，找右 DataFrame 中 key 匹配的所有行，配对组成新行。
  • 左或右中找不到匹配的行 —— 默认全部消失（inner join）。
  • 如果一边的 key 出现多次，会与另一边所有匹配的行各自配对（笛卡尔积）。

4. ⭐ 预测 merge 后的行数（最重要！）

核心算法：
  1. 列出 left 和 right 中 key 列的所有唯一值。
  2. 对同时出现在两边的每个 key 值 k：数 left 中有几行（L_k），数 right 中有几行（R_k），贡献 L_k × R_k 行。
  3. 只在一边出现的 key 值 → 贡献 0 行（消失）。
  4. 把所有 key 值的贡献相加 = 总行数。

👇 完整 worked example（lec9 经典示例）

offer_percentage:                clothes:
  clothing_type  offer            item   retail_price
0  Shirt          20            0  Dress  150
1  Pants          30            1  Shirt   30
2  Dress          50            2  Shoes   90
3  Shorts         30            3  Pants   50
4  Shoes          50            4  Shoes   70

# offer_percentage.merge(clothes, left_on='clothing_type', right_on='item')

逐 key 计数：

key 值left 出现right 出现贡献
Shirt111 × 1 = 1
Pants111 × 1 = 1
Dress111 × 1 = 1
Shorts101 × 0 = 0（消失）
Shoes121 × 2 = 2（一行配两行）
合计5 行

5. Index-based merge

当 key 是 index 而不是列时，用 left_index=True 或 right_index=True：

# offers_by_item 的 index 是 'clothing_type'，clothes 中是 'item' 列
offers_by_item.merge(clothes, left_index=True, right_on='item')

如果一个 DataFrame 是按 index 来 merge 的（set_index 之后），但你想让它的 index 也变回普通列再 merge —— 先 .reset_index() 再 .merge(...)。

⚠️ 常见陷阱与考点

陷阱 1：默认 inner join，不匹配的行直接消失
左 DataFrame 中 key 在右边找不到 → 那行不出现在结果里。所以 merge 后行数可能比左边少。
例：Winter 2026 Q8.1 中，pizza10 的 10 个 kind 中只有 6 个出现在 notes 里 → 结果 6 行（不是 10）。
陷阱 2：重复 key 导致行数膨胀（笛卡尔积）
一边 key 出现 a 次，另一边出现 b 次 → 结果有 a × b 行。
极端例：左有 1 行 key=Shoes，右有 2 行 key=Shoes → 结果 2 行（左边那行被复制配两次）。
陷阱 3：on= vs left_on=/right_on=
on='col' 要求两边都有同名列。如果两边列名不同（如 'clothing_type' vs 'item'），必须用 left_on=/right_on=。
陷阱 4：dtype 不匹配 → 静默地变成空 DataFrame
例如左边 key 列是 int（92109），右边是 str（"92109"）—— pandas 不会报错，merge 结果是 0 行（或全是 NaN）。
陷阱 5：merge 时 index 会丢失
如果 left 是按 'word' 设了 index 的，left.merge(right, ...) 之后 'word' 这个 index 不再是 index，结果的 index 变成 0,1,2,...。
解决：merge 前用 .reset_index() 把它变成普通列，merge 后再 .set_index('word')。
例：Fall 2025 Q2.1。
陷阱 6：多列 groupby 后必须 .reset_index() 才能像普通列一样使用
没 reset 时，结果的 'section' 和 'first' 是 MultiIndex 的一部分，不能用 .get('section') 取出。
顺序无关性：A.merge(B, ...) 和 B.merge(A, ...) 结果内容相同，只是行/列顺序可能不同。考试中如果问"两个表达式是否等价"——只看内容、不看顺序时是等价的。

📝 往年真题

每题都涉及多列 groupby 或 merge。重点练 merge 行数预测。

题 1 — Fall 2024 Q9（merge 行数预测，最经典款）

treat 有 200 行，含 15 种不同的 candy。trick 以 "candy" 为 index，有 25 行：其中 15 行对应 treat 中出现的 candy，另外 10 行是 treat 中没有的 candy。

trick_or_treat = trick.merge(treat, left_index=True, right_on="candy")

问：trick_or_treat 有几行？
选项：15 / 25 / 200 / 215 / 225 / 3000 / 5000

显示答案与解析

答案：200

逐 key 计数：

  • trick 共 25 个 candy，每个出现 1 次（因为 candy 是 trick 的 index，唯一）。
  • 这 25 个中：15 个在 treat 中也出现，10 个不在。
  • treat 共 200 行，分布在 15 种 candy 上。设第 i 种 candy 在 treat 中出现 n_i 次（Σn_i = 200）。
  • 对那 15 个共有 candy：每个贡献 1 × n_i。求和 = Σn_i = 200。
  • 那 10 个只在 trick 出现的 candy：贡献 0（inner join 消失）。
  • 总和 = 200。

中文解析：本质上 trick 是个"价格表"（每种糖一行），merge 之后相当于给 treat 的每行打上价格标签。所以行数 = treat 的行数 = 200。10 个 trick 中独有的 candy 在 treat 里没有匹配，全部消失。

题 2 — Fall 2024 Q13（多列 groupby 嵌套解读，extra credit）
double = treat.groupby("candy").count().groupby("address").count()
# 已知 double.loc[1].get("how_many") 等于 5

下列哪些是有效解读？

  1. 有 5 户人家各自只发 1 种糖。
  2. 有 5 种糖各自只在 1 户人家发。
  3. 只有 1 户人家发 5 种糖。
  4. 只有 1 种糖在 5 户人家发。
显示答案与解析

答案：1 和 2 都对

解析：

  • 第一步 treat.groupby("candy").count() → 每行一种 candy，"address" 列存"该 candy 出现在多少户家"，"how_many" 列存同样的数（count 把所有非聚合列填一样的 count）。
  • 第二步 .groupby("address").count() → 把上面 DataFrame 按 "address" 列（其值是"该 candy 出现的户数"）分组。
  • .loc[1] 即 "address" = 1 的那行 → 这些 candy 都"只出现在 1 户人家"。
  • .get("how_many") 等于 5 → 这样的 candy 共有 5 种。
  • 所以："有 5 种糖各自只在 1 户人家发" ✓ —— 选项 2 直接对。
  • 选项 1 也等价：5 种糖各自只在 1 户出现 ⇔ 那 5 户各自从这角度看只发了这一种糖（这道题给的是 valid interpretation 而不是唯一）。注意官方答案两个都选。

关键 takeaway：嵌套 groupby 解读看：第一次 group 后每行代表什么，第二次 group 又把"代表"变成"分类标签"。

题 3 — Winter 2025 Q5.2（用 merge 算 total cost）

面包菜单：Sourdough $2 / Whole Wheat $3 / Multigrain $4。eaten 是一个 numpy array，记录每片面包的种类。请用 merge 算总价。

available = bpd.DataFrame().assign(
    Type = ["Sourdough", "Whole Wheat", "Multigrain"]
).assign(Cost = [2, 3, 4])
consumed = bpd.DataFrame().assign(Eaten = eaten)
combined = available.merge(consumed, left_on = __(d)__,
                                     right_on = __(e)__)
total_cost = combined.__(f)__
显示答案与解析

答案：(d) "Type"; (e) "Eaten"; (f) get("Cost").sum()

中文解析：

  • 核心思路：consumed 中每片面包要"查表"得到价格 → 用 merge 把价格贴到 consumed 的每一行上。
  • 左 key = available 的 "Type"，右 key = consumed 的 "Eaten"。
  • 对于 consumed 中每一行（一片面包），available 中正好有 1 行 type 匹配（available 是去重菜单）→ merge 后每片面包对应 1 行，行数 = len(eaten)。
  • 每行已带上对应的 Cost → 把 Cost 列加起来就是总价。

对比 5.1（for-loop 法）：循环每种面包，np.count_nonzero(eaten == bread) * price 累加。两种方法都要会！

题 4 — Spring 2025 Q5.1 / 5.2（merge 写法 + 行数预测）

tariffs 有 50 行，index 是 0..49，其中一列叫 "Country"。trade_partners 以 "Country" 为 index。tariffs 中所有国家都在 trade_partners 中，但 trade_partners 中有些国家不在 tariffs。

5.1：写一行代码 merge tariffs 和 trade_partners。
5.2：merged 有几行？

显示答案与解析

5.1 答案（写法之一）：

merged = tariffs.merge(trade_partners, left_on="Country", right_index=True)

因为 tariffs 中 Country 是普通列，trade_partners 中 Country 是 index。

5.2 答案：50

逐 key 计数：

  • tariffs 中每个 Country 出现 1 次（共 50 个）。
  • trade_partners 中每个 Country 也只出现 1 次（因为是 index，唯一）。
  • tariffs 的所有 Country 都在 trade_partners 里 → 每个 country 贡献 1 × 1 = 1 行。
  • 50 个 country × 1 = 50 行。
  • 注意：trade_partners 中那些不在 tariffs 的 country，inner join 时直接消失。
题 5 — Spring 2025 Q5.3（groupby 后 Proportion 是否仍 sum 到 1？）

原 trade_partners 的 "Proportion" 列加起来 = 1（所有进口比例之和）。下面哪些 DataFrame 中 "Proportion" 加起来仍是 1？

  1. trade_partners
  2. trade_partners.groupby("Continent").mean()
  3. trade_partners.groupby("Continent").sum()
  4. merged
显示答案与解析

答案：1 和 3

  • (1) ✓ 题目给定。
  • (2) ✗ 各 continent 的平均 proportion 之和不一定等于 1。
  • (3) ✓ 各 continent 的总 proportion 之和 = 所有国家的 proportion 之和 = 1。
  • (4) ✗ merged 是 inner join，丢掉了 trade_partners 中不在 tariffs 的国家 → proportion 之和 < 1。

关键 takeaway：merge 之后 proportion 之和"漏水"是个常见考点。

题 6 — Spring 2025 Q5.4（按 group 画 median 图）

写一行代码，画图展示每个大洲的 median reciprocal tariff。

显示答案与解析 
merged.groupby("Continent").median().plot(kind="barh", y="Reciprocal Tariff")

解析：

  • 需要先在 merged 上做（不在 tariffs 上做，因为 tariffs 没有 Continent 列）。
  • 按 "Continent" 分组 → 取 .median() → 画 kind="barh"（或 "bar"）的条形图，y 指定 tariff 列。
题 7 — Fall 2025 Q2.1 / 2.2（reset_index/set_index 包夹 merge + 行数预测）

words 以 "word" 为 index，有 5357 行。tags 是默认整数 index，有 36 行（每行一个 part-of-speech tag）。

merged = words.reset_index().merge(__(a)__).set_index("word")

2.1：填空 (a) —— 想让 "ps"（words 中 tag 列）和 tags 中 tag 匹配。
2.2：merged 有几行？

显示答案与解析

2.1 答案：tags, left_on="ps", right_on="tag"

2.2 答案：5357

逐 key 计数：

  • 每个 word 在 words 中出现 1 次，每行有一个 ps（tag）。
  • tags 中每个 tag 唯一（36 个 tag，每个 1 行）。
  • words 中所有 ps 必然是 36 个 tag 之一（题目说"All values in 'ps' are tags from Penn Treebank system"）。
  • 所以每个 word 都能在 tags 中找到 1 个匹配 → 每行贡献 1 × 1 = 1。
  • 总和 = 5357 行。

为什么要 reset_index/set_index 包夹：merge 会丢掉 index。先 reset 把 word 变成普通列保住，merge 后再 set_index('word') 恢复。

题 8 — Fall 2025 Q5.1（多列 groupby 行数预测）
grouped = (words.assign(
              diff_category = words.get("diff").apply(classify_difficulty),
              length_category = words.get("length").apply(classify_length))
           .groupby(["diff_category", "length_category"])
           .count()
           .reset_index())

已知：classify_difficulty 输出 "easy"/"moderate"/"hard"（3 类），classify_length 输出 "short"/"medium"/"long"（3 类）。题目假设每种组合至少有一个 word。

问 grouped 有几行？

显示答案与解析

答案：9 行

解析：多列 groupby 产生每个唯一组合一行。

  • diff_category 有 3 个值，length_category 有 3 个值 → 共 3 × 3 = 9 种组合。
  • 题目说每种组合至少有一个 word → 9 种组合全出现 → 9 行。
  • 如果题目没说"每种组合至少一个"，结果只能 ≤ 9，没法精确算。
题 9 — Winter 2026 Q8.1（key 不完全重合的 merge）

pizza10 有 10 行（10 种 pizza kind，每种 1 行），notes 有 8 行（index 是 kind，每个 kind 1 行）。题目说："some kinds in pizza do not appear in notes, and some kinds in notes do not appear in pizza"。看前面 notes 的定义，notes 的 8 个 kind 是: Cheese, Pepperoni, Veggie, Margherita, Hawaiian, Supreme, BBQ Chicken, White。pizza10 的 10 个 kind 是: Cheese, Pepperoni, Supreme, Hawaiian, Cheese, Pepperoni, Margherita, Cheese, Veggie, Bamboo。

pizza10.merge(notes, left_on="kind", right_index=True).shape[0]

选项：0 / 6 / 9 / 10 / Cannot be determined

显示答案与解析

答案：9

逐 key 计数（pizza10 中每行的 kind 是否在 notes 的 index 里）：

pizza10 行kind在 notes 中？贡献
0Cheese是1
1Pepperoni是1
2Supreme是1
3Hawaiian是1
4Cheese是1
5Pepperoni是1
6Margherita是1
7Cheese是1
8Veggie是1
9Bamboo否0

合计：9 + 0 = 9 行。

关键：notes 中独有的 kind（如 BBQ Chicken, White）不影响左边来主导的 merge —— 但它们仍然消失（inner join）。pizza10 中独有的 Bamboo 也消失。

题 10 — Spring 2024 Q6.1（按 index merge，部分重合）

charlie 有 172 行（联系人），norah 有 88 行。其中 12 个联系人两边共有，160 个仅 charlie 独有，76 个仅 norah 独有。两个表都以联系人 ID 为 index（每个 ID 唯一）。

charlie.merge(norah, left_index=True, right_index=True).shape[0]
显示答案与解析

答案：12

逐 key 计数：每个 ID 在 charlie 中出现 ≤ 1 次，在 norah 中出现 ≤ 1 次（index 唯一）。

  • 共有的 12 个 ID：每个贡献 1 × 1 = 1，共 12 行。
  • 仅 charlie 独有的 160 个：在 norah 找不到 → 贡献 0。
  • 仅 norah 独有的 76 个：在 charlie 找不到 → 贡献 0。
  • 合计 12 行。

记忆口诀：两边 key 都唯一时，inner join 行数 = 两边 key 集合的交集大小。

题 11 — Spring 2024 Q6.2（key 重复时的笛卡尔积）

norah 不小心把那 12 个共有联系人复制了一份 → norah 现在 100 行（76 独有 + 12×2 共有的重复）。问：norah.merge(norah, left_index=True, right_index=True).shape[0] 是多少？

显示答案与解析

答案：76 + 12 × (2 × 2) = 76 + 48 = 124 行

逐 key 计数（这是 norah 自己 merge 自己）：

  • 76 个独有 ID：每个在 norah 中出现 1 次 → 1 × 1 = 1，共 76 行。
  • 12 个被复制的 ID：每个在 norah 中出现 2 次 → 2 × 2 = 4，共 12 × 4 = 48 行。
  • 合计 = 76 + 48 = 124 行。

陷阱：很多人会错答 100（norah 的行数）。重点是每个重复 ID 自配自变成 2×2=4 行，不是 2 行！
